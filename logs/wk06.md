## Objective: Learning return values and recursion in Python

### Return Values
- Learned that some functions return values that can be assigned to variables or used in expressions
- Understood that built-in functions like `abs` and `math.sqrt` return results instead of printing them
- Learned how to write a function that returns a value with the `return` statement
- Discovered that local variables inside a function, like `area`, cannot be accessed from outside the function

### None and Pure Functions
- Learned that functions without a `return` statement return `None`
- Understood the difference between a function that prints something and a function that returns something
- Learned what a **pure function** is: a function that only returns a value and has no side effects
- Realized that assigning the result of a printing function still gives `None`

### Return Values and Conditionals
- Learned that every possible path through a function with return statements needs to end with a return value
- Discovered that missing a return on one branch can cause a function to return `None`
- Learned that code after a `return` statement is dead code because it can never run
- Understood why dead code usually means something about the logic needs to be checked

### Incremental Development
- Learned how to build a function in small steps and test after each change
- Used temporary variables like `dx`, `dy`, and `sum_of_squares` to debug intermediate results
- Learned that scaffolding code is useful during development but should be removed later
- Understood that incremental development reduces debugging time and makes errors easier to isolate

### Boolean Functions
- Learned how to write functions that return `True` or `False`
- Understood that boolean functions are useful for encapsulating tests like divisibility
- Learned that comparisons already produce booleans, so extra `== True` checks are unnecessary

### Recursion with Return Values
- Learned how recursion works with base cases and recursive cases
- Understood the factorial definition and how it maps directly into Python
- Learned the idea of a **leap of faith**: assume the recursive call works and reason about the rest of the function
- Discovered that recursive functions must have a base case or they can recurse forever

### Type Checking and Input Validation
- Learned that recursion can fail if the input type is wrong, even when a base case exists
- Used `isinstance` to check that an argument is an integer before recursing
- Learned that negative inputs should be checked separately
- Understood that validating function inputs helps prevent infinite recursion and bad results

### Debugging Recursive Functions
- Learned to debug by checking the function’s inputs, outputs, and how the caller uses the return value
- Used print statements as scaffolding to show the flow of recursive calls
- Realized that debugging recursive code is easier when each function is broken into checkpoints
- Understood that a good debug strategy is to ask whether the problem is in the arguments, the function body, or the caller

### Key Takeaways
- Return values are different from printed output
- Pure functions are easier to test and reason about
- Incremental development is one of the best ways to avoid debugging overload
- Recursion works well when the base case, recursive case, and input validation are all correct
